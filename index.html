How It Works

The sync cycle runs every 5 minutes automatically. A full cycle completes in under 30 seconds for the entire fleet.

Solarman API
→
Sync Engine
→
MongoDB
→
Dashboard

Every 5 min → normalize → upsert → broadcast via SSE to all connected clients

Sync Pipeline — Step by Step

1
Fetch Plant List

GET /station/search

Discovers all registered stations in the Solarman account.

2
Real-Time Per Plant

POST /station/realtime

Pulls current generation, battery SOC, grid power, load for each plant.

3
Device Inventory

POST /device/page

Enumerates all inverters, dataloggers, and meters per station.

4
Inverter Telemetry

POST /device/currentData

Pulls per-device metrics: voltage, current, power, temperature, alarms.

5
Normalize & Map

Unifies disparate field names, units, and response shapes into a single canonical schema before storage.

6
Upsert & Broadcast

Writes to MongoDB (idempotent upserts) + pushes live updates to all connected dashboards via SSE.

Technical Safeguards

Making this look simple meant solving these real problems under the hood.

ChallengeHow We Solved ItSeverity
Solarman tokens expire silently Multi-source auto-extraction pipeline with graceful fallback Critical
API rate limits (15 req/min) Intelligent request queuing + staggered dispatch High
Response shapes vary across endpoints Normalizer maps all variants to a single canonical model High
Network timeouts & interruptions Retry with exponential backoff + partial success persistence Moderate
Device timestamps drift Server-side time anchoring Moderate
Duplicate records from repeated syncs Idempotent upsert logic (safe to run multiple times) Moderate
Live + history must coexist In-memory cache for live, indexed queries for history Moderate
One plant failure blocks others Per-plant error isolation + graceful degradation Moderate
